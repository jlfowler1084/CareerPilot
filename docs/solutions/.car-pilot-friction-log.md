# CAR-181 Pilot — Running Friction Log

## Phase 0 — gitleaks smoke test allowlist surprise

**Phase:** 0 (pre-flight)
**Friction:** The prescribed smoke test (`echo 'AWS="AKIAIOSFODNN7EXAMPLE"'`) returned exit 0 with "no leaks found", contrary to the prompt's expectation of "nonzero exit + findings". Initial read: gitleaks might be broken.
**Root cause:** `AKIAIOSFODNN7EXAMPLE` is the canonical AWS docs example key and is in gitleaks' built-in allowlist. `command -v`-style checks wouldn't catch this; the key is structurally valid but intentionally ignored to avoid false positives in tutorials/READMEs.
**Workaround:** Used `AKIAABCDEFGHIJ123456` as the smoke-test key instead — exit 1, "leaks found: 1". Gitleaks is working correctly.
**v1.1 proposal:** Update the CAR/INFRA pilot contract templates to use a non-allowlisted synthetic key in the smoke test step so the test is unambiguous. A comment noting WHY the example key is intentionally skipped would also help future coordinators avoid confusion.

## Phase 4 Stream A — worktree missing node_modules and .env.local

**Phase:** 4A (Stream A Phase B build step)
**Friction:** The Next.js `npm run build` step inside the worktree failed until the subagent installed `node_modules` and copied `.env.local` from the main working directory. Git worktrees share the `.git` directory but NOT the `node_modules/` directory or any gitignored files.
**Root cause:** `node_modules/` is gitignored and therefore not present in a freshly created worktree. `.env.local` is also gitignored. Both are required for `next build` to succeed. The contract didn't include a "pre-build setup" step.
**Workaround:** Subagent ran `npm install` in the worktree's dashboard directory and copied `.env.local`. Added ~1-2 min of overhead.
**v1.1 proposal:** Add a "worktree setup" step to dashboard-touching contracts: `cd dashboard && npm install && cp ../../.env.local .` (or symlink). This is deterministic and should be templated in the contract generator so frontend streams don't have to discover it at runtime.

## Phase 4 Stream D — checkpoint protocol ill-fitted for a serial tail stream

**Phase:** 4D (Stream D dispatch)
**Friction:** The checkpoint protocol is designed for parallel streams where a coordinator review gate prevents one stream from drifting into another's files mid-flight. Stream D was inherently serial — it ran after all other streams had already merged, and its Phase A files (new files only) had zero overlap risk with any live stream. The standard two-phase checkpoint pattern added process overhead for a situation it wasn't designed for.
**Workaround:** Dispatched D directly to "Phase A + Phase B combined" — the coordinator still ran the 6-step review when D returned, but the checkpoint stop was folded into the final return rather than a mid-flight pause. This saved one round-trip.
**v1.1 proposal:** The Parallelization Map should classify each stream's checkpoint discipline as either "parallel-safety required" (standard checkpoint) or "serial-tail" (combined return acceptable). Tail streams queued behind a shared-interface freeze are serial at execution time and don't benefit from the mid-flight gate.
